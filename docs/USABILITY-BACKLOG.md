# Бэклог юзабилити (раунд 2) — взгляд опытного пользователя

Прошлый бэклог (`UX-BACKLOG.md`) был про **консистентность и полировку**.
Этот — про **скорость реальных сценариев**: я прошёл систему как регистратор
и врач (создание приёма, поиск, расписание, отчёты) и отмечал, где теряется
время или внимание. Приоритет — по частоте в рабочем дне.

Легенда: 🔎 замечено вживую · 💭 из опыта, стоит проверить/сделать.

---

## P1 — трение в КАЖДОМ приёме (самое частое действие)

### R1. Свёрнутые сводки для заполненных секций «Владелец»/«Животное» · 🔎 M
**Наблюдение.** В форме приёма секции 1 «Владелец» и 2 «Животное» после выбора
всё равно занимают по целой карточке. На планшете (1280×800) секции 1–3
заполняют экран, а секция 4 «Услуги и препараты» — **главное денежное
действие** — уходит под сгиб и открывается свёрнутой. Врач каждый раз
прокручивает мимо уже выбранного владельца/животного к услугам.
**Предложение.** После выбора сворачивать секции 1 и 2 в одну строку-сводку
(«КД Ковалёв Дмитрий · 🐶 Рекс, Собака» + «изменить»), автоматически
разворачивать «Услуги». Итог по вертикали ↓, услуги поднимаются в первый экран.

### R2. Врач по умолчанию — текущий пользователь · 🔎 S
**Наблюдение.** Поле «Врач» открывается как «— не указан —». Врач, ведущий
свой приём, обязан выбирать себя каждый раз; забыл — приём выпадает из
отчёта по врачу и сводки выработки.
**Предложение.** Если у вошедшего есть связанный сотрудник (staff_id) —
подставлять его врачом по умолчанию (с возможностью сменить). Регистратор
без staff_id видит прежнее «не указан».

### R3. Быстрые пресеты для «Следующий приём» · 🔎 S
**Наблюдение.** «Следующий приём» — голое поле даты (дд.мм.гггг). Повторный
осмотр назначают постоянно и типовыми интервалами; ввод даты вручную на
планшете медленный.
**Предложение.** Рядом чипы-пресеты: +7 дней · +2 недели · +1 мес · +3 мес ·
+6 мес · +1 год (клик подставляет дату от даты приёма). Это же поле кормит
напоминания бота и отчёт «Предстоящие».

## P2 — заметное трение в ежедневной работе

### R4. Enter/клавиатура выбирает единственное совпадение · 💭 S
**Наблюдение.** В поиске владельца и в поиске услуги результат надо кликать
мышью/пальцем. Когда совпадение одно — Enter должен его выбирать; стрелки
↑/↓ + Enter — двигаться по списку. Ускоряет ввод и удобно со сканером/клавой.
**Предложение.** Клавиатурная навигация в автокомплитах владельца, животного,
услуги и глобального поиска (Enter — выбрать выделенное/единственное).

### R5. Расписание нескольких врачей — колонки, а не только фильтр · 🔎 M
**Наблюдение.** День расписания — одна колонка слотов 08:00–19:30; несколько
врачей смотрятся только переключением фильтра по одному. В клинике с 2+
врачами не видно общей загрузки дня.
**Предложение.** Режим «по врачам»: колонки рядом (врач = столбец) на широком
экране; на планшете — компактная группировка/цвет по врачу в одной ленте.

### R6. Мягкое предупреждение о приёме без врача · 💭 S
**Наблюдение.** Есть предупреждение о приёме без позиций (сделано ранее), но
приём без врача сохраняется молча — и потом теряется в отчёте по врачу.
**Предложение.** Тот же мягкий диалог при сохранении: «Врач не указан — приём
не попадёт в отчёт по врачу. Сохранить всё равно?». Снимается автоматически,
если R2 подставляет врача.

## P3 — сглаживание углов

### R7. Явное «Обновить данные с сервера» · 🔎 S
**Наблюдение.** Столкнулся с ситуацией (реальной для клиники после
восстановления сервера из бэкапа — см. заметку про планшет): локальная база
расходится с сервером, а единственный путь обновиться — «Сбросить локальные
данные» в Настройках → Данные, что звучит пугающе-деструктивно.
**Предложение.** Отдельная неразрушающая кнопка «Обновить с сервера»
(полный pull) рядом со статусом синхронизации — без формулировки «сбросить».

### R8. Строка позиции: Enter добавляет следующую · 💭 S
**Наблюдение.** Каждую позицию добавляешь кнопкой «+ Добавить позицию». При
вводе нескольких услуг подряд это лишние клики.
**Предложение.** Enter в последнем поле строки (или после выбора услуги из
автокомплита) добавляет новую пустую строку и ставит фокус в неё.

### R9. Быстрый «Новый приём» из карточки владельца · 💭 S
**Наблюдение.** Из строки животного «Новый приём» есть (в меню «⋯»). Из
карточки владельца путь к приёму для его питомца менее очевиден.
**Предложение.** Кнопка «Новый приём» в карточке владельца (и в карточке
животного) с предзаполненными владельцем/животным — минус несколько шагов
на частом сценарии «пришёл повторно».

## Что работает хорошо (не трогать)
- Глобальный поиск в шапке: «Барс» → «Барсик · Иванова Мария» мгновенно.
- Закреплённый «Итог К оплате» в футере формы приёма (виден при прокрутке).
- Автокомплит владельца находит по префиксу фамилии и по телефону.
- Инлайн-создание нового владельца/животного прямо в форме приёма.
- Чипирование: кликабельные счётчики-фильтры, поиск под сканер (inputmode).

## Порядок
Начать с **R1 + R2 + R3** — они бьют по самому частому действию (приём) и
вместе ощутимо сокращают путь «пришёл клиент → сохранён приём». Затем R4–R6.
