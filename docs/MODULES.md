# Модульная архитектура — план

Цель: включать функционал **помодульно по необходимости** (телеграм, кабинет
владельца, склад, будущие). Модуль = самодостаточная единица: своя схема,
синк-сущности, маршруты, экраны, настройки, роли, флаг вкл/выкл.

Сейчас опциональность точечная: флаг склада (`server_settings.warehouse_enabled`
+ `GET /settings/modules`), телеграм — по наличию токена. Портал/чипирование/
вакцинации всегда включены. Нужно превратить это в **систему**.

## Принципы

1. **Тумблер — рантайм, не пересборка.** Вкл/выкл меняет только поведение
   (маршруты, меню, синк), не схему.
2. **Данные не удаляются при выключении.** Таблицы модуля создаются всегда
   (миграции идемпотентны). Выключение прячет UI и отклоняет эндпоинты;
   включение возвращает всё как было. (Так уже работает склад.)
3. **Инкрементально, без big-bang.** Strangler: ядро на месте, модули
   выносим по одному под общий контракт. Первый пилот — телеграм.
4. **Ядро минимально.** В ядре — то, без чего клиника не работает.

## Что ядро, что модуль

**Ядро (всегда вкл):** auth/пользователи, синк-движок, владельцы, животные,
приёмы, каталог позиций, дашборд, оболочка настроек, глобальный поиск.

**Модули (опционально):**
- `telegram` — бот, напоминания, журнал доставки.
- `portal` — кабинет владельца (портал по телефону + одноразовый пароль).
  Зависит от `telegram` для доставки паролей (или будущего SMS-модуля).
- `warehouse` — склад (уже вынесен по духу; привести к контракту).
- Кандидаты позже: `chipping` (чипирование), `vaccinations`, `scheduling`
  (расписание), `reports` (пакет отчётов). Пока они «ядро-ish» — выносим,
  когда контракт обкатан.

## Контракт модуля

Каждый модуль **декларирует** (а не разбрасывает по файлам):
- `Key` (напр. "warehouse") и человекочитаемое имя.
- `Migrations()` — DDL (CREATE TABLE IF NOT EXISTS / ALTER идемпотентно).
- `SyncEntities()` — список синкаемых сущностей (см. ниже).
- `Routes(mux)` — HTTP-маршруты; регистрируются всегда, но внутри — гейт
  `if !a.moduleEnabled("warehouse") { 404 }`.
- `Enabled(app)` — читает флаг.
- `Roles()` — какие роли/права добавляет (склад → роль `warehouse`).
- `DependsOn()` — зависимости (portal → telegram).
- Фронт-манифест: nav-пункты, страницы (init-функции), вкладки настроек,
  гейт по флагу.

## Бэкенд: реестр модулей

Ввести лёгкий интерфейс и центральный реестр вместо трёх мест регистрации.

```go
type Module interface {
    Key() string
    Migrations() []string
    SyncEntities() []SyncEntity
    RegisterRoutes(mux *http.ServeMux, a *app)
    Roles() []string
    DependsOn() []string
}

var moduleRegistry = []Module{ /* telegramModule{}, portalModule{}, warehouseModule{} */ }
```

- **Миграции**: `runMigrations` дополнительно проходит по
  `moduleRegistry[*].Migrations()`. Схема ядра остаётся в `schema`.
- **Маршруты**: `setupRoutes` вызывает `m.RegisterRoutes(mux, a)` для каждого
  модуля. Внутри хендлеров — гейт `a.moduleEnabled(key)` → 404/403 при выкл.
- **Флаги**: единый `a.moduleEnabled(key)` = `getSetting(key+"_enabled", def)`.
  `GET /settings/modules` возвращает карту всех модулей; `PUT /settings/module/{key}`
  (админ) переключает + проверяет зависимости (нельзя включить portal без telegram).
- **Роли**: `validRoles` собирается из ядра + `m.Roles()`.

## Синк: перевести на табличный реестр (ключевой энейблер)

Сейчас синк — **по-сущностный руками** (регистрация в 4 местах:
`syncPushPayload`/`syncPullData` структуры, merge, pull, canPush). Это главный
барьер модульности: добавить сущность = править ядровые файлы.

Ввести описание сущности данными:

```go
type SyncEntity struct {
    Table       string            // "stock_movements"
    Columns     []string          // список колонок
    PermTable   string            // "warehouse" (для canPush)
    ScanRow     func(*sql.Rows) (map[string]any, error)
    // merge/pull генерятся из Table+Columns (upsert ON CONFLICT + version-first)
}
```

- Обобщённые `pushEntity`/`pullEntity` работают по `SyncEntity` (текущая
  логика conflict-resolution уже единообразна — `clientWinsVersion`).
- Push/pull payload становится `map[string][]record` вместо фиксированных
  полей — модуль добавляет сущность, не трогая структуры ядра.
- Клиентские `ENTITY_STORES`/`STORE_ORDER` тоже строятся из манифестов
  модулей (см. фронт). `DB_VERSION` растёт при добавлении стораджа.

Это самый крупный рефактор, но он окупается: после него **новый модуль с
данными не трогает ядро синка вообще**.

## Фронтенд: реестр модулей

Сейчас всё в одном `pages.js` (~9.8k строк). План:

1. **Разбить `pages.js` на файлы** по областям: `core/*` (владельцы, животные,
   приёмы, дашборд) и `modules/telegram.js`, `modules/portal.js`,
   `modules/warehouse.js`. Каждый — свой `<script>`, регистрируется в общий
   реестр (не ломая текущий IIFE-стиль — просто несколько IIFE).
2. **Манифест модуля на фронте**:
   ```js
   VetModules.register({
     key: "warehouse",
     nav: [{ group: "warehouse", page: "warehouse", label: "Склад", icon: "box" }],
     pages: { warehouse: initWarehouse },
     settingsTab: { label: "Склад", render: renderWarehouseSettings },
     stores: ["warehouses", "stock_movements"],   // для db.js/sync.js
     enabledFlag: "warehouse",
   });
   ```
3. **Гейт по флагу**: `refreshModules()` (уже есть) читает `/settings/modules`,
   показывает/прячет nav и вкладки настроек модулей, кэширует офлайн.
4. **Роутер** (`init(page)` map и `validPages`) собирается из манифестов, а не
   хардкодится.

## Семантика вкл/выкл

| Аспект | Поведение |
|---|---|
| Данные | Сохраняются (таблицы всегда есть). Выкл — только гейт. |
| Маршруты | При выкл возвращают 404/403. |
| Меню/экраны | Скрыты; роутер не пускает на страницу модуля. |
| Синк | Продолжает синкать (данные консистентны); можно скипать для экономии. |
| Роли | Роль модуля (`warehouse`) остаётся валидной; вход такого юзера при выключенном модуле — показать заглушку/логаут. |
| Зависимости | Нельзя включить portal без telegram (проверка в PUT). |

## Поэтапный план (strangler, без остановки продукта)

**Фаза 0 — единые флаги (малый риск, сразу польза).**
`moduleEnabled(key)` + `GET /settings/modules` для всех модулей + тумблеры в
Настройках (телеграм/портал/склад). Портал получает флаг (сейчас всегда вкл).

**Фаза 1 — реестр модулей на бэке (без смены синка).**
Интерфейс `Module`, `moduleRegistry`; маршруты и миграции регистрируются
через него. Хендлеры телеграма/портала/склада заворачиваются в гейт
`moduleEnabled`. `validRoles` из модулей.

**Фаза 2 — табличный синк (ключевой рефактор).**
Обобщённые push/pull по `SyncEntity`; payload → карта. Мигрируем сущности по
одной (сначала складские — они новые и изолированы), ядровые — последними.

**Фаза 3 — фронт-реестр.**
Разбить `pages.js`; `VetModules.register`; nav/router/настройки из манифестов;
db.js/sync.js берут стораджи из манифестов.

**Фаза 4 — вынести существующие модули под контракт.**
Пилот — `telegram` (изолирован, есть настройки, малый blast-radius). Затем
`portal`, затем `warehouse` (уже близок). Каждый — отдельный PR/коммит,
обратимо.

**Фаза 5 (опц.) — выделить «ядро-ish» в модули** (chipping, vaccinations,
scheduling, reports), когда контракт обкатан.

## Пилот: телеграм как первый модуль

Малый риск, чёткие границы: `telegram.go` уже отдельный, настройки уже в UI.
Обернуть в `telegramModule` (Migrations нет — таблицы уже в ядре; Routes —
/settings/telegram*; Enabled — по токену ИЛИ явному флагу; фронт-манифест —
вкладка настроек). Отрабатываем контракт на нём, потом тиражируем.

## Что НЕ делать

- Не дробить ядро (владельцы/животные/приёмы) на модули — они всегда нужны,
  дробление добавит сложности без пользы.
- Не выносить в микросервисы — это один бинарник offline-first, сила в
  простоте деплоя (`prod/start.bat`). Модульность — внутри монолита
  («модульный монолит»), не распределённая система.
- Не удалять данные при выключении модуля.
